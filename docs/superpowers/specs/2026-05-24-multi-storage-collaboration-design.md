# 多数据库协作与对象存储改造设计

## 1. 背景

当前项目后端采用 `FastAPI + SQLAlchemy + MySQL`，生产侧只有一个关系型数据库入口，上传文件直接写入 `backend/uploads` 本地目录。这个实现方式在项目早期足够简单，但已经出现三个明显阻碍：

1. 文件存储与应用进程强耦合。
2. 大体积作品文件和历史资料持续增长后，本地磁盘、备份、迁移与跨机部署成本会快速升高。
3. 教师批量下载全班作品时，当前实现仍以本地文件目录为前提，不适合作为后续扩容的长期基础。

与此同时，仓库中仍保留“测试依赖 SQLite 内存库”的现状，这本身没有问题，但如果团队把“测试用 SQLite”与“生产可继续混用 SQLite 文件”混在一起理解，会进一步干扰后续架构演进。

本次设计目标不是迁移到 Django，也不是一次性引入多个运行中关系库，而是在保持现有 FastAPI 主体不变的前提下，建立一套可持续扩展的“多数据库/多存储协作”方案。

## 2. 目标

本次设计确认的首期目标如下：

1. 保持现有 `FastAPI + SQLAlchemy + MySQL` 主业务架构不变。
2. 以 `MySQL` 继续承载核心事务数据，作为当前唯一运行中的业务主库。
3. 引入兼容 S3 协议的对象存储，首选 `SeaweedFS`，用于承载新上传的文件资源。
4. 对历史已经位于本地 `uploads` 目录的文件保持兼容，不要求首期一次性迁移。
5. 后端建立统一文件存储抽象层，使业务层不再直接依赖本地磁盘路径。
6. 为后续 `PostgreSQL` 承接作品扩展属性、搜索分析或非结构化元数据预留边界，但首期不将其接入运行链路。
7. 明确 SQLite 在本项目中的定位：仅允许作为测试内存库，不再保留或引入任何业务运行时 SQLite 文件。

## 3. 非目标

本次设计明确不做以下事项：

1. 不迁移框架到 Django 或 DRF。
2. 不在首期将 PostgreSQL 纳入生产读写路径。
3. 不重写当前教师端或学生端页面结构。
4. 不改变统一响应格式 `{"code": 0, "data": ..., "message": "ok"}`。
5. 不在首期完成历史文件全量搬迁。
6. 不在本次方案中引入复杂消息队列、异步任务平台或独立文件处理微服务。

## 4. 现状分析

### 4.1 数据库现状

当前数据库接入点主要位于：

- `backend/app/core/config.py`
- `backend/app/db/session.py`

当前只有一个 `DATABASE_URL`，默认指向 MySQL。代码结构上没有为“多数据源协作”预留显式配置分层，也没有独立的文件元数据表。

### 4.2 文件存储现状

当前上传接口位于：

- `backend/app/api/v1/routes/upload_routes.py`

主要特征如下：

1. 文件由接口一次性读入内存后校验。
2. 通过 UUID 重命名后写入 `backend/uploads`。
3. 返回给前端的是 `/uploads/<filename>` 相对访问路径。
4. 业务表中保存的是文件 URL，而不是可迁移的存储元数据。

### 4.3 批量下载现状

当前教师批量下载位于：

- `backend/app/api/v1/routes/teacher_routes.py`

当前实现依赖：

1. 从 `Project.report_url` 反推本地文件名。
2. 再从 `backend/uploads` 中读取物理文件。
3. 使用 `io.BytesIO + zipfile.ZipFile` 在内存中构造 ZIP。
4. 最后通过 `StreamingResponse` 返回。

这里虽然已经使用了 `StreamingResponse`，但压缩包本身仍先构建在内存对象中，随着文件规模变大，依然存在明显的内存上限风险。

### 4.4 SQLite 现状

当前确认到的 SQLite 使用分为两类：

1. 正常保留：`backend/tests/` 中的 SQLite 内存库，用于测试隔离。
2. 不应继续扩散：任何业务运行时 `.db/.sqlite/.sqlite3` 文件。

从当前仓库扫描结果看，没有发现项目业务使用中的 SQLite 数据文件；需要清理的重点不是“删除大量现存 sqlite 文件”，而是：

1. 删除未来可能误提交的 SQLite 文件。
2. 统一文档和配置认知，避免开发过程再次引入运行时 SQLite。
3. 保留测试用内存 SQLite，不将其误伤。

## 5. 方案对比与选型

### 方案 A：维持 MySQL + 本地文件系统

优点：

- 改动最少。
- 研发成本最低。

缺点：

- 无法解除当前文件存储阻碍。
- 不利于扩容、备份、迁移与多机部署。
- 批量下载和大文件场景的风险继续累积。

结论：不采用。

### 方案 B：MySQL + SeaweedFS（S3）+ PostgreSQL 预留

优点：

- 先解决当前最迫切的文件存储与下载问题。
- 不打断现有 FastAPI 事务模型。
- 未来可按模块把扩展元数据逐步引入 PostgreSQL。
- 实施节奏可控，回滚成本较低。

缺点：

- 需要补一层文件元数据与存储适配。
- 上传、访问、批量下载逻辑都要做兼容改造。

结论：首期采用。

### 方案 C：MySQL + PostgreSQL + 对象存储一次性上齐

优点：

- 中长期结构完整。

缺点：

- 当前项目复杂度会立刻上升。
- 首期要同时处理双关系库边界、一致性、运维与排障问题。
- 与当前“先解阻碍、再平滑演进”的目标不匹配。

结论：暂不采用。

## 6. 目标架构

首期目标架构如下：

```text
前端
  -> FastAPI 路由层
    -> 业务服务层
      -> MySQL（核心事务数据）
      -> 文件存储服务抽象层
           -> LocalStorageAdapter（兼容历史本地文件）
           -> S3StorageAdapter（SeaweedFS，新上传文件）
```

架构原则：

1. 业务表继续以 MySQL 为唯一事务真源。
2. 文件内容不再以“业务表中的 URL 字符串”作为唯一事实来源。
3. 新文件写入对象存储，旧文件通过本地适配器继续可读。
4. 访问、下载、删除等动作统一通过存储服务层完成。

## 7. 数据与存储分层设计

### 7.1 MySQL 的职责

MySQL 在首期继续保存：

1. 用户、教师、学生、班级、课程、章节、题目、公告、作品等核心业务数据。
2. 文件与业务对象之间的引用关系。
3. 文件元数据记录。

不再让 MySQL 直接承担的职责：

1. 文件二进制内容存储。
2. 通过业务 URL 反推物理磁盘路径的唯一定位方式。

### 7.2 SeaweedFS 的职责

SeaweedFS 承担：

1. 课程资料文件。
2. 学生作品报告。
3. 图片、压缩包、视频等较大资源。
4. 后续新增多媒体素材。

选型理由：

1. 协议兼容 S3，便于使用成熟 Python SDK。
2. 对小文件与大文件都较友好。
3. 协议和维护策略相对适合长期教育场景。

### 7.3 PostgreSQL 的预留职责

首期不接入运行链路，但预留如下方向：

1. 学生作品扩展属性。
2. 灵活 JSON 结构查询。
3. 后续统计分析、搜索、标签聚合等新模块。

预留原则：

1. 先在设计中定义边界，不在首期代码中提前接入无实际用途的第二关系库。
2. 后续只有在出现真实查询或建模需求时，再为特定模块新增 PostgreSQL 会话与模型。

## 8. 文件元数据设计

### 8.1 新增文件元数据表

建议在 `backend/app/models/entities.py` 中新增统一文件元数据模型，例如 `StoredFile`。首期字段建议如下：

```text
id                主键
biz_type          业务类型，如 material/project_report/project_image
biz_id            业务对象 ID，可为空，允许先上传后绑定
storage_provider  存储提供者，取值 local / s3
bucket_name       对象存储桶名，本地存储可为空
object_key        对象键；本地文件则保存相对路径
original_name     原始文件名
stored_name       实际存储文件名
content_type      检测后的 MIME 类型
extension         文件扩展名
size_bytes        文件字节数
sha256            内容摘要，用于校验与去重预留
status            active / deleted / orphan
created_by        上传者 ID
created_at        创建时间
```

### 8.2 与现有业务表的关系

首期不建议大规模重写现有表结构，可以采用“兼容演进”：

1. `Material.url`
2. `Project.report_url`
3. `Project.image_url`
4. `ProjectImage.image_url`

这些旧字段暂时继续保留，用于前端兼容和历史记录读取。

同时新增可选关联字段，例如：

1. `materials.file_id`
2. `projects.report_file_id`
3. `projects.cover_file_id`
4. `project_images.file_id`

演进策略：

1. 新上传文件优先写 `StoredFile`。
2. 新业务记录同时写入 `file_id`，并兼容生成旧 `url` 字段。
3. 老记录如果只有旧 URL，没有 `file_id`，则走历史兼容路径。

这样可以避免一次性改坏当前前后端联调路径。

## 9. 存储适配层设计

### 9.1 抽象目标

在 `backend/app/services/` 或 `backend/app/core/` 中建立统一文件存储服务，例如：

```text
StorageService
  - save_upload(...)
  - open_stream(...)
  - build_public_url(...)
  - exists(...)
  - delete(...)
```

### 9.2 两类适配器

#### 本地适配器

职责：

1. 继续读取 `backend/uploads` 中的历史文件。
2. 允许后端对旧记录执行存在性检查、下载与打包。

注意：

1. 首期不再作为默认新文件写入目标。
2. 仅用于历史兼容和迁移过渡。

#### S3 适配器

职责：

1. 处理所有新上传文件保存。
2. 对接 SeaweedFS S3 网关。
3. 提供对象读取流和对象删除能力。

建议配置项：

```text
STORAGE_BACKEND=s3
S3_ENDPOINT=
S3_ACCESS_KEY=
S3_SECRET_KEY=
S3_BUCKET_PUBLIC=
S3_BUCKET_PRIVATE=
S3_REGION=
S3_FORCE_PATH_STYLE=true
```

### 9.3 URL 策略

为了减少前端改动，首期不建议把对象存储真实地址直接散落到业务表中。建议：

1. 对外仍由后端提供统一访问 URL。
2. 前端继续消费项目域内 URL。
3. 后端根据 `file_id` 或旧 URL 决定从本地还是对象存储取文件。

建议新增统一文件访问路由，例如：

```text
GET /api/files/{file_id}
```

用途：

1. 统一鉴权与访问控制。
2. 屏蔽底层存储位置。
3. 为未来切换 CDN、签名链接、私有桶下载保留空间。

## 10. 上传安全设计

当前 `upload_validation.py` 只校验扩展名与大小。首期改造时应补强为“扩展名 + MIME/魔数”双重校验。

### 10.1 校验要求

1. 不能只信任前端文件扩展名。
2. 在写入对象存储或本地磁盘前，必须校验真实文件类型。
3. 对压缩包、PDF、图片、视频分别使用明确的魔数或 MIME 识别。
4. 若识别失败，直接返回用户可理解错误提示。

### 10.2 存储隔离要求

1. 文件命名使用随机名，不保留用户原始文件名作为物理对象名。
2. 原始文件名仅作为元数据保存。
3. 即使仍保留本地兼容目录，也不应放在可执行目录或可被误解释为脚本的路径下。
4. 禁止未来继续在业务逻辑中通过用户传入文件名直接拼路径。

## 11. 教师批量下载设计

这是本次方案中的关键工程点。

### 11.1 当前问题

当前批量下载依赖本地文件路径，并在内存中构造完整 ZIP，随着作品压缩包、视频或大报告数量上升，会产生明显 OOM 风险。

### 11.2 首期目标

首期要改成“流式读取 + 临时文件打包 + 流式响应”的稳定实现，而不是继续在 `BytesIO` 中累积完整 ZIP。

### 11.3 推荐实现

后端打包过程建议如下：

1. 根据作品记录取出 `file_id` 或旧 `report_url`。
2. 通过 `StorageService.open_stream()` 取得统一输入流。
3. 将每个文件按块写入 ZIP 归档。
4. ZIP 输出目标不放在纯内存 `BytesIO`，而使用临时文件或可溢写缓冲策略。
5. 压缩完成后通过 `FileResponse` 或生成器方式返回。

### 11.4 首期取舍

为了降低实现风险，首期不强求“边压缩边实时向浏览器输出最终 ZIP 字节流”。  
首期允许采用更稳妥的方案：

1. 服务器侧临时文件打包。
2. 打包完成后流式下载。
3. 下载结束后清理临时文件。

这样虽然不是最极致的实时流式 ZIP，但可以显著降低内存风险，并且比当前实现更符合现有项目的可维护性。

## 12. 历史文件兼容与迁移设计

### 12.1 首期兼容策略

首期采用“双路径兼容”：

1. 新文件：写入 SeaweedFS，对应生成 `StoredFile` 记录。
2. 老文件：继续从 `backend/uploads` 读取。
3. 业务读取：优先走 `file_id`；没有 `file_id` 时回退到旧 URL 解析。

### 12.2 后续迁移策略

后续单独提供迁移脚本，逐步把历史文件转入对象存储：

1. 扫描引用本地 `/uploads/...` 的业务记录。
2. 校验本地文件存在。
3. 上传到对象存储。
4. 创建 `StoredFile`。
5. 回填业务表 `file_id`。
6. 保留旧 `url` 字段用于兼容。

迁移完成后，可再评估是否下线本地历史兼容路径。

## 13. SQLite 残留清理策略

本项需要明确“清理什么、不清理什么”。

### 13.1 应清理

1. 仓库根目录或后后端目录未来若出现的业务运行时 `.db/.sqlite/.sqlite3` 文件。
2. README、设计文档、启动文档中任何暗示“生产或开发主流程可直接使用 SQLite 文件”的描述。
3. 本地脚本、初始化脚本中与业务运行时 SQLite 相关的默认配置。
4. `.gitignore` 中未覆盖的 SQLite 文件模式。

### 13.2 不应清理

1. `backend/tests/` 里的 SQLite 内存测试用法。
2. 第三方依赖中的类型定义文件或示例文件。
3. 与当前任务无关的缓存目录，除非它们被错误纳入版本管理。

### 13.3 实际判断

根据当前扫描结果，本仓库暂未发现业务使用中的 SQLite 数据文件，因此“去除 SQLite 文件”应被解释为：

1. 先从规范和配置上禁止继续生成。
2. 若后续再发现业务 sqlite 文件，再做定向清理。

不应为了满足文字要求而误删测试或依赖文件。

```text
DATABASE_URL=...
```

首期仍只有这一条进入运行链路。

### 14.2 存储配置

```text

S3_ACCESS_KEY=...
```

### 14.3 PostgreSQL 预留配置

可在设计中预留但不要求首期启用：

```text

```

要求：

1. 首期代码不得因为这个变量缺失而启动失败。
2. 只有后续真实接入 PostgreSQL 时才读取并初始化。

## 15. 接口影响

### 15.1 尽量不变的部分

以下现有业务接口尽量保持返回结构稳定：

1. 通用上传接口响应外层结构。
2. 教师资料管理接口。
3. 学生作品上传接口。
4. 教师批量下载接口。

### 15.2 允许增加的字段

在不破坏前端的前提下，接口可以新增：

1. `file_id`
2. `storage_provider`
3. `content_type`
4. `size_bytes`

原则：

1. 新字段只增不删。
2. 老字段 `url` 先保留。

## 16. 实施边界

本次设计后的首期实施范围建议控制为：

1. 新增文件元数据模型与兼容 schema。
2. 新增存储配置与存储抽象层。
3. 改造上传接口，使新上传走对象存储。
4. 改造作品与资料的文件引用方式，支持 `file_id + 旧 url` 双读。
5. 改造教师批量下载，去掉内存整包 `BytesIO` 方案。
6. 补测试、补文档、补 `.gitignore` 和启动说明。

不进入首期实施的内容：

1. 历史文件全量迁移。
2. PostgreSQL 实际接线。
3. CDN、签名下载链接、跨区域复制。
4. 异步导出中心。

## 17. 风险与应对

### 风险 1：前端仍依赖旧 URL 语义

应对：

1. 首期保留 `url` 字段。
2. 后端统一生成兼容访问路径。

### 风险 2：历史记录没有 `file_id`

应对：

1. 保留旧 URL 回退路径。
2. 后续再用迁移脚本逐步补齐。

### 风险 3：对象存储不可用时上传失败

应对：

1. 明确错误提示。
2. 不自动悄悄回退到本地写入，避免制造新的双写混乱。
3. 在开发环境允许显式切换 `STORAGE_BACKEND=local` 用于联调。

### 风险 4：批量下载临时文件清理不彻底

应对：

1. 所有临时压缩文件进入固定临时目录。
2. 请求完成后主动清理。
3. 增加异常分支清理与定期巡检策略。

## 18. 验收标准

首期实施完成后，至少满足以下验收标准：

1. 新上传的课程资料、作品报告、作品图片默认写入 SeaweedFS，而不是 `backend/uploads`。
2. 历史仍指向 `/uploads/...` 的记录依然可访问。
3. 新旧文件均可被教师批量下载功能正确打包。
4. 批量下载不再依赖完整内存 `BytesIO` 构建整包 ZIP。
5. 上传接口对扩展名和真实文件类型均做校验。
6. 项目文档中不再把 SQLite 描述为业务运行时数据库选项。
7. `backend/tests/` 的 SQLite 内存测试仍然可用。
8. 前端构建通过，受影响的教师端页面可正常打开，无乱码、无空白。

## 19. 建议的后续演进顺序

建议后续按以下顺序推进：

1. 先完成本设计对应的实施计划。
2. 首期落地 `MySQL + SeaweedFS + 本地历史兼容`。
3. 观察一轮上传、下载、资料管理和作品提交流程。
4. 再决定是否编写历史文件迁移脚本。
5. 当出现真实复杂查询需求后，再单独设计 PostgreSQL 子方案。

## 20. 结论

本项目当前不适合直接升级为“多运行中关系库并行读写”的复杂形态。更稳妥的路径是：

1. 继续把 MySQL 作为唯一核心事务库。
2. 通过 SeaweedFS 接管新文件二进制存储。
3. 用统一文件元数据与存储抽象层解除业务对本地目录的直接依赖。
4. 用双路径兼容保护历史文件。
5. 把 PostgreSQL 放在后续确有需要时再接入，而不是提前引入复杂度。

这条路径能够直接解除当前“文件存储阻碍后续开发”的核心问题，同时保持与现有 FastAPI 代码结构、测试体系和教师端接口的连续性。
