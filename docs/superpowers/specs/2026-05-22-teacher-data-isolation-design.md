# 教师数据隔离权限设计

## 背景

当前教师端多个核心数据仍是全局共享的：班级、课程、章节、资料、题目等数据没有统一归属教师字段。结果是任意教师登录后都可能看到其它教师创建或维护的数据。这不符合正常教学平台的权限边界。

## 目标

1. 每个教师只能管理自己创建或归属自己的班级、课程、章节、资料和题目。
2. 教师端“学生数据”只显示该教师班级中的学生。
3. 教师端“资料管理”只显示该教师的课程、章节和资料。
4. 教师端“题库管理”只显示该教师的题目和可用章节。
5. 任务、公告继续沿用已有 `teacher_id` 隔离逻辑，并校验公告选择的班级属于当前教师。
6. 学生端只接收自己所在班级对应教师发布的任务；课程学习内容可以先保留公开可见，后续再做“按班级发布课程”的细粒度控制。
7. 旧库数据兼容：历史缺少教师归属的数据默认分配给测试教师 `T001`，避免升级后页面突然空白。

## 非目标

1. 不引入管理员角色。
2. 不做组织、学院、教研室等多级租户模型。
3. 不重写前端页面结构。
4. 不改变统一响应格式。
5. 不做实时同步或 WebSocket。

## 数据模型设计

新增教师归属字段：

```text
classes.teacher_id       -> users.id
courses.teacher_id       -> users.id
chapters.teacher_id      -> users.id
materials.teacher_id     -> users.id
questions.teacher_id     -> users.id
```

字段类型使用 `String(32)`，允许旧库兼容阶段先默认 `T001`。新增数据必须写入当前登录教师 ID。

关系原则：

1. 班级属于教师。
2. 课程属于教师。
3. 章节属于教师，并且章节归属的课程必须属于同一教师。
4. 资料属于教师，并且资料归属的章节必须属于同一教师。
5. 题目属于教师，并且题目归属的章节必须属于同一教师。
6. 学生账号本身不是某个教师私有，但教师只能通过自己的班级看到学生数据。

## 接口隔离设计

### 班级

`/api/classes` 下所有教师端接口按当前教师过滤：

- 列表只返回当前教师班级。
- 创建班级写入 `teacher_id=current_user.id`。
- 删除、查看学生、添加学生、移除学生都必须校验班级属于当前教师。
- Excel 导入学生时，自动创建的班级归属当前教师。

### 学生数据

`GET /api/teacher/students`：

- 不传 `class_id`：返回当前教师所有班级下的学生。
- 传 `class_id`：只返回当前教师该班级学生；如果班级不属于当前教师，返回 404 或空结果。

### 课程、章节、资料、题目

所有教师端管理接口写入并过滤当前教师：

- `/api/questions/courses`
- `/api/chapters`
- `/api/materials`
- `/api/questions`

学生端读取学习内容时暂不强制按教师隔离，仍可读取已发布课程内容。后续如果要课程按班级或教师发布，再增加课程发布关系表。

### 公告和任务

公告已有 `teacher_id`，本次补充校验：

- 教师创建公告时，`class_id` 必须属于当前教师。
- 教师查看任务完成报告时，公告必须属于当前教师。

## 旧数据兼容

`backend/app/db/schema_compat.py` 扩展为：

1. 检查上述表是否存在 `teacher_id`。
2. 不存在则 `ALTER TABLE ADD COLUMN teacher_id VARCHAR(32) DEFAULT 'T001'`。
3. 对已有空值执行 `UPDATE ... SET teacher_id='T001' WHERE teacher_id IS NULL OR teacher_id=''`。

测试库通过 SQLAlchemy 建表会直接包含字段；旧 MySQL 库通过兼容脚本补齐。

## 验收标准

1. 教师 A 创建的班级，教师 B 登录后看不到。
2. 教师 A 创建的课程、章节、资料、题目，教师 B 登录后看不到。
3. 教师 A 的学生数据页只显示教师 A 班级中的学生。
4. 教师不能向不属于自己的班级发布公告。
5. 教师不能编辑、删除不属于自己的课程、章节、资料和题目。
6. 旧库启动后不报缺字段错误，历史数据默认归属 `T001`。
7. 后端测试和前端构建通过。
