# 部署扩容方案设计

## 1. 背景

当前项目后端采用 `FastAPI + SQLAlchemy + MySQL`，以单进程 `uvicorn --reload` 方式运行，数据库连接池使用 SQLAlchemy 默认值（pool_size=5, max_overflow=10），前端通过 Vite 开发服务器代理请求，无 Nginx 反向代理层，无 Redis 缓存层。

项目计划部署到学校环境，预期使用规模为：
- **几百到几千学生**
- **几十到几百教师**
- **峰值并发用户数 500–2000**

当前架构在开发和小规模演示（几十人）下没有问题，但不具备直接承载千人级并发的能力。需要在部署形态和关键配置上做针对性改造。

## 2. 目标

1. 使当前架构能够稳定支撑 **500–2000 峰值并发用户**。
2. 改动以部署配置和少量代码调整为主，不做架构级重构。
3. 保持现有 `FastAPI + SQLAlchemy + MySQL` 技术栈不变。
4. 不引入微服务、Kubernetes、读写分离等超出当前规模的复杂度。
5. 方案可分阶段实施，每阶段完成后即可独立验证效果。

## 3. 非目标

1. 不做多机集群部署（单机或双机即可满足当前规模）。
2. 不引入 WebSocket/SSE 实时推送（轮询在当前规模下足够）。
3. 不做数据库读写分离（单机 MySQL 可支撑当前负载）。
4. 不引入异步任务队列（Celery/RQ 等）。
5. 不做 CDN 部署（学校内网环境，单台 Nginx 带宽足够）。
6. 不改动前端业务逻辑和页面结构。

## 4. 现状瓶颈分析

### 4.1 应用进程 — 单进程是硬天花板

`backend/main.py` 中的启动方式：

```python
uvicorn.run("main:app", host="0.0.0.0", port=8050, reload=True)
```

单进程处理所有请求。FastAPI 虽然是异步框架，但当前路由使用同步 SQLAlchemy Session（`get_db` 是同步生成器），每个数据库操作会阻塞事件循环，由线程池接管执行。默认线程池大小约 40，并发超过 40 后请求开始排队。

### 4.2 数据库连接池 — 默认值过小

`backend/app/db/session.py` 中未显式配置连接池：

```python
engine = create_engine(settings.database_url, echo=False, pool_pre_ping=True)
```

SQLAlchemy 默认 `pool_size=5`，`max_overflow=10`，总计最多 15 个并发连接。500 并发用户下，大部分请求会因等待数据库连接而超时。

### 4.3 无缓存层 — 每次请求穿透到数据库

章节列表、课程信息、题目内容等低频变化数据，每次请求都直接查询 MySQL。这些数据被大量用户重复读取，造成不必要的数据库压力。

### 4.4 无反向代理 — 开发模式直接暴露

当前生产部署没有 Nginx 层，Vite 代理仅在开发模式生效。缺少：
- 静态资源托管与 gzip 压缩
- 请求缓冲与负载分发
- 连接复用（keep-alive）
- 限流与安全防护基础

### 4.5 前端缺乏高延迟适配

- axios 超时仅 10 秒，高并发下后端响应变慢容易误触发超时
- 无全局 loading 状态，用户在请求等待期间无反馈
- 无请求失败重试机制

## 5. 方案设计

### 5.1 第一阶段：部署形态改造（必须做）

这是投入产出比最高的阶段，完成即可支撑 2000 并发。

#### 5.1.1 多进程启动

将单进程 `uvicorn.run()` 改为多 worker 模式。

两种方案任选其一：

**方案 A：Uvicorn 多 worker（推荐，改动最小）**

```bash
uvicorn main:app --host 0.0.0.0 --port 8050 --workers 4
```

**方案 B：Gunicorn + Uvicorn Worker**

```bash
gunicorn main:app -w 4 -k uvicorn.workers.UvicornWorker --bind 0.0.0.0:8050
```

建议 worker 数量 = CPU 核数 × 2 + 1（4 核机器用 4-9 个 worker）。

**注意事项：**
- `reload=True` 不能与多 worker 共存，生产环境必须去掉
- 多 worker 下 `lifespan` 中的 seed 逻辑需要加分布式锁或只在单个 worker 执行，避免多个 worker 同时跑 seed
- 需要新增一个生产启动脚本（如 `start_prod.sh` 或更新 `start_backend.bat`）

#### 5.1.2 MySQL 连接池调优

修改 `backend/app/db/session.py`，显式配置连接池参数：

```python
engine = create_engine(
    settings.database_url,
    echo=False,
    pool_pre_ping=True,
    pool_size=30,        # 基础连接池大小
    max_overflow=20,     # 允许额外创建的连接数
    pool_recycle=3600,   # 连接最大存活时间（秒），避免 MySQL wait_timeout 断连
    pool_timeout=30,     # 获取连接的最长等待时间
)
```

**同时需要调整 MySQL 侧配置：**
- `max_connections` 建议设为 200-300（默认 151 偏低）
- `wait_timeout` 建议设为 28800（8 小时），与 `pool_recycle` 配合

连接池总容量 = workers × pool_size = 4 × 30 = 120，加上 overflow 的 80，总计最多 200 个连接，在 MySQL `max_connections=300` 范围内留有余量。

#### 5.1.3 Nginx 反向代理

新增 Nginx 配置文件 `deploy/nginx.conf`，核心配置：

```nginx
upstream backend {
    server 127.0.0.1:8050;
    keepalive 32;
}

server {
    listen 80;
    server_name _;

    # 前端静态资源
    location / {
        root /path/to/frontend/dist;
        try_files $uri $uri/ /index.html;
        expires 7d;
        gzip on;
        gzip_types text/css application/javascript application/json image/svg+xml;
    }

    # API 反向代理
    location /api/ {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_connect_timeout 5s;
        proxy_read_timeout 60s;
        proxy_buffering on;
    }

    # 文件访问
    location /uploads/ {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_read_timeout 120s;
    }
}
```

**Nginx 带来的收益：**
- 静态资源由 Nginx 直接返回，不经过 Python 进程
- gzip 压缩减少 60-70% 的传输体积
- keep-alive 连接复用减少 TCP 握手开销
- 请求缓冲保护后端不受慢客户端影响

### 5.2 第二阶段：Redis 缓存层（强烈建议）

在第一阶段到位后，如果发现 MySQL 负载仍然偏高，或者响应时间不达预期，加这一层。

#### 5.2.1 需要缓存的数据

| 数据 | 变化频率 | 建议 TTL | 预期效果 |
|------|---------|---------|---------|
| 章节列表 | 极低（教师偶尔新增） | 1 小时 | 减少 80%+ 的章节查询 |
| 课程列表 | 极低 | 1 小时 | 减少 80%+ 的课程查询 |
| 题目内容 | 低（教师偶尔增删改） | 30 分钟 | 减少 70%+ 的题目查询 |
| 教师工作台统计 | 中 | 5 分钟 | 减少 90%+ 的聚合查询 |
| 用户信息 | 低 | 30 分钟 | 减少鉴权时的用户表查询 |

#### 5.2.2 实现方式

- 新增 `backend/app/services/cache_service.py`，封装 Redis get/set/delete 操作
- 在各 service 层的查询方法中加入缓存逻辑：先查缓存，命中则返回，未命中则查数据库并回写缓存
- 数据变更时（教师新增章节、修改题目等）主动清除相关缓存
- Redis 配置项加入 `backend/.env`：`REDIS_URL=redis://127.0.0.1:6379/0`

#### 5.2.3 依赖

- `backend/requirements.txt` 增加 `redis` 包
- 服务器安装 Redis（Windows 可用 Memurai 或 WSL）

### 5.3 第三阶段：前端适配（小修补）

这些改动不影响业务逻辑，只改善高延迟下的用户体验。

#### 5.3.1 axios 配置调整

修改 `frontend/src/api/http.ts`：

- `timeout` 从 10000ms 调整为 30000ms
- 增加请求失败自动重试一次的逻辑（仅对 GET 请求和网络错误重试，不对 4xx 业务错误重试）

#### 5.3.2 全局 Loading 状态

在 `App.vue` 或路由守卫中增加全局请求计数器：
- 有进行中的请求时显示顶部进度条（可用 NProgress 或 Element Plus 的 Loading）
- 所有请求完成后隐藏

#### 5.3.3 生产构建配置

`frontend/vite.config.ts` 中确认：
- `build.target` 设为合适的浏览器范围
- 产物由 Nginx 托管（已在 5.1.3 中覆盖）
- Vite dev proxy 仅开发环境使用，生产不走 Vite

## 7. 验收标准

### 第一阶段验收

1. 后端以 4 个 worker 进程运行，`ps` 或任务管理器可见多个 uvicorn worker 进程。
2. `backend/app/db/session.py` 中连接池参数已显式配置，`pool_size >= 20`。
3. MySQL `max_connections >= 200`。
4. Nginx 正常代理 `/api/` 和 `/uploads/` 到后端，前端静态资源由 Nginx 直接托管。
5. 使用压测工具（如 `locust` 或 `wrk`）模拟 500 并发用户，所有接口响应时间 P95 < 2 秒，无 5xx 错误。
6. 去掉 `reload=True`，生产启动不自动重载代码。

### 第二阶段验收

1. Redis 服务正常运行，后端能成功连接。
2. 章节列表、课程列表接口在缓存命中时响应时间 < 50ms。
3. 教师新增章节后，缓存自动清除，学生端能查到新章节。
4. 压测 1000 并发用户，MySQL 实际活跃连接数 < 50（缓存拦截了大部分查询）。

### 第三阶段验收

1. 后端响应 3-5 秒时，前端不报超时错误。
2. 请求进行中时页面有 loading 反馈。
3. 网络瞬断后 GET 请求自动重试一次成功。
4. `npm run build` 构建产物正常，Nginx 托管后页面可访问。

## 8. 风险与应对

### 风险 1：多 worker 下 seed 逻辑重复执行

**表现：** 4 个 worker 同时启动，都检测到空数据库并执行 seed，导致重复插入或主键冲突。

**应对：** 在 `lifespan` 中增加基于数据库行锁或文件锁的互斥逻辑，确保只有一个 worker 执行 seed。或者将 seed 从 `lifespan` 中移出，改为独立的初始化脚本，在部署时手动执行一次。

### 风险 2：多 worker 下内存中的全局状态丢失

**表现：** 如果有代码使用了模块级变量做缓存或计数器，多 worker 下各进程独立，状态不共享。

**应对：** 当前项目没有发现模块级可变状态。如果后续发现，应迁移到 Redis 或数据库中。

### 风险 3：Windows 部署环境的限制

**表现：** `uvicorn --workers` 在 Windows 下使用 `spawn` 而非 `fork`，内存开销略大；Redis Windows 原生版本已停止维护。

**应对：** 生产环境建议使用 Linux 服务器部署。如果必须在 Windows 下运行，可使用 Memurai（Redis 的 Windows 兼容版本）或通过 WSL2 运行 Redis。

### 风险 4：Nginx 配置错误导致文件上传失败

**表现：** 大文件上传时被 Nginx 的 `client_max_body_size` 默认值（1MB）拦截。

**应对：** 在 Nginx 配置中设置 `client_max_body_size 50m`，与后端 `MAX_UPLOAD_SIZE` 保持一致。

## 9. 扩展上限参考

| 并发用户数 | 需要的最低配置 | 说明 |
|-----------|--------------|------|
| 100 以下 | 单 worker，无 Nginx，无 Redis | 当前架构直接可用 |
| 100–500 | 2 workers + 连接池调优 | 第一阶段的部分改动 |
| 500–2000 | 4 workers + 连接池 + Nginx | 第一阶段全部改动 |
| 2000–5000 | 第一阶段 + Redis 缓存 | 第一阶段 + 第二阶段 |
| 5000–10000 | 以上 + 读写分离 / 多机部署 | 超出当前方案范围，需重新设计 |

当前项目目标定位在 **500–2000 并发**，第一阶段改动即可满足。
