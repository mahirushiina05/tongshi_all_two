# 部署扩容方案 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 使当前架构稳定支撑 500–2000 峰值并发用户，改动以部署配置和少量代码调整为主，不做架构级重构。

**Architecture:** 保持 `FastAPI + SQLAlchemy + MySQL` 技术栈，分三个阶段实施：第一阶段改造部署形态（多进程 + 连接池 + Nginx），第二阶段引入 Redis 缓存层，第三阶段做前端高延迟适配。每阶段可独立验收。

**Tech Stack:** FastAPI、Uvicorn/Gunicorn、SQLAlchemy、MySQL、Redis（第二阶段）、Nginx、Vue 3、Vite、Element Plus、Axios

---

## Phase 1：部署形态改造（必须做）

> 完成后即可支撑 2000 峰值并发。

### Task 1.1：改造后端多进程启动

修改启动方式，从单进程改为多 worker。

- [ ] 修改 `backend/main.py`：去掉 `reload=True`，或将 reload 仅在非生产环境启用。确保 `lifespan` 中的 seed 逻辑在多 worker 下安全（增加互斥判断或改为独立初始化脚本）。
- [ ] 新增 `backend/start_prod.sh`（Linux）或更新 `start_backend.bat`（Windows），使用 `uvicorn main:app --host 0.0.0.0 --port 8050 --workers 4` 启动。
- [ ] 验证：启动后在终端中看到 4 个 worker 进程日志，`ps aux | grep uvicorn`（Linux）或任务管理器中可见多个进程。

### Task 1.2：调优 MySQL 连接池

显式配置 SQLAlchemy 连接池参数。

- [ ] 修改 `backend/app/db/session.py`：增加 `pool_size=30, max_overflow=20, pool_recycle=3600, pool_timeout=30`。
- [ ] 修改 MySQL 配置（`my.cnf` 或 `my.ini`）：设置 `max_connections=300`，`wait_timeout=28800`。
- [ ] 验证：启动后端，用 `SHOW STATUS LIKE 'Threads_connected';` 确认连接池正常工作。压测时活跃连接数应在预期范围内（4 worker × 30 pool_size = 120 基础连接）。

### Task 1.3：配置 Nginx 反向代理

建立 Nginx 配置，托管前端并代理后端请求。

- [ ] 新增 `deploy/nginx.conf`：配置 upstream backend（指向 127.0.0.1:8050），location `/api/` 和 `/uploads/` 代理到后端，location `/` 托管前端 `dist` 目录。
- [ ] 配置 `client_max_body_size 50m`（与后端上传限制一致）。
- [ ] 配置 gzip 压缩：`gzip on; gzip_types text/css application/javascript application/json image/svg+xml;`。
- [ ] 配置静态资源缓存：`expires 7d;`。
- [ ] 前端执行 `npm run build`，确认 `dist` 目录产物正常。
- [ ] 验证：通过 Nginx 端口访问前端页面正常，API 请求正常代理到后端，上传下载功能正常。

### Task 1.4：第一阶段压测验证

- [ ] 使用压测工具（推荐 locust：`pip install locust`，编写简单压测脚本）模拟 500 并发用户。
- [ ] 验收标准：所有接口 P95 响应时间 < 2 秒，无 5xx 错误，MySQL 活跃连接数 < 100。
- [ ] 如有条件，进一步压测到 1000-2000 并发，确认系统稳定。

## Phase 2：Redis 缓存层（强烈建议）

> 第一阶段运行后，观察 MySQL 负载和响应时间。如果数据库查询压力大或响应偏慢，实施此阶段。

### Task 2.1：引入 Redis 依赖与配置

- [ ] 修改 `backend/requirements.txt`：增加 `redis` 包。
- [ ] 修改 `backend/app/core/config.py`：增加 `REDIS_URL` 配置项，默认值 `redis://127.0.0.1:6379/0`。
- [ ] 服务器安装 Redis（Linux 直接 `apt install redis-server`；Windows 使用 Memurai 或 WSL2）。
- [ ] 验证：`redis-cli ping` 返回 `PONG`。

### Task 2.2：实现缓存服务

- [ ] 新增 `backend/app/services/cache_service.py`：封装 Redis 连接池初始化、get（反序列化 JSON）、set（序列化 JSON + TTL）、delete、delete_pattern（按前缀批量清除）操作。
- [ ] 提供装饰器或上下文管理器，简化 service 层的缓存调用模式：`with cache_or_query(cache_key, ttl, query_func)`。

### Task 2.3：在核心 service 中接入缓存

- [ ] 修改 `backend/app/services/course_service.py`：课程列表查询结果缓存 1 小时，课程创建/修改时清除缓存。
- [ ] 修改 `backend/app/services/chapter_service.py`：章节列表查询结果缓存 1 小时，章节创建/修改时清除缓存。
- [ ] 修改 `backend/app/services/question_service.py`：题目列表查询结果缓存 30 分钟，题目增删改时清除缓存。
- [ ] 修改 `backend/app/services/teacher_service.py`：工作台统计数据缓存 5 分钟，涉及数据变更时清除缓存。

### Task 2.4：第二阶段压测验证

- [ ] 压测 1000 并发用户，观察 Redis 命中率（`redis-cli INFO stats` 中的 `keyspace_hits` / `keyspace_misses`），目标命中率 > 70%。
- [ ] 验收标准：MySQL 实际活跃连接数 < 50，章节/课程接口响应时间 < 50ms（缓存命中时）。

## Phase 3：前端高延迟适配（可选）

> 改善后端响应变慢时的用户体验，不影响功能正确性。

### Task 3.1：调整 axios 超时与重试

- [ ] 修改 `frontend/src/api/http.ts`：`timeout` 从 10000 改为 30000。
- [ ] 增加响应拦截器中的重试逻辑：仅对 GET 请求和网络错误（无 response）重试一次，不对 4xx 业务错误重试。使用 `config.__retryCount` 标记，避免无限重试。

### Task 3.2：增加全局 Loading 状态

- [ ] 在 `frontend/src/App.vue` 或新建 `frontend/src/composables/useLoading.ts` 中实现请求计数器：axios 请求拦截器 +1，响应拦截器 -1，计数 > 0 时显示顶部进度条。
- [ ] 可使用 Element Plus 的 `ElLoading` 或引入轻量的 `nprogress` 库。

### Task 3.3：前端生产构建验证

- [ ] 执行 `npm run build`，确认无类型错误和构建失败。
- [ ] 通过 Nginx 访问构建产物，验证所有页面正常加载、路由跳转正常、API 调用正常。

## 验证与上线清单

全部阶段完成后，按此清单确认：

- [ ] 后端以 4 个 worker 运行，无 `reload=True`。
- [ ] MySQL 连接池 `pool_size=30`，`max_connections=300`。
- [ ] Nginx 反向代理正常，前端静态资源由 Nginx 托管。
- [ ] Redis 正常运行（如实施了第二阶段），缓存命中率 > 70%。
- [ ] 压测 500 并发：P95 < 2s，无 5xx。
- [ ] 压测 1000 并发（如实施了第二阶段）：P95 < 3s，MySQL 活跃连接 < 50。
- [ ] 前端 `npm run build` 通过，Nginx 托管后所有页面可正常访问。
- [ ] 上传 50MB 文件成功（Nginx `client_max_body_size` 和后端限制一致）。
- [ ] 教师批量下载功能正常（与多存储改造配合验证）。

## 风险检查

- [ ] 多 worker 下 seed 逻辑不重复执行（加互斥或改为独立脚本）。
- [ ] 无模块级可变全局状态（多 worker 下各进程独立，不会出现状态不一致）。
- [ ] Windows 部署时确认 `uvicorn --workers` 正常工作（使用 spawn 模式，内存开销可接受）。
- [ ] Nginx 配置中 `proxy_read_timeout` 足够长（教师批量下载大文件需要 60-120 秒）。
